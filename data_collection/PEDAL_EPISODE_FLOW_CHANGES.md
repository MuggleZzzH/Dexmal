# Pedal Episode Flow Changes

本文档说明 `data_collection/main.py` 相对原始版本的主要修改点。

## 1. 原始版本的采集流程

原始脚本是单次采集模型，流程比较简单：

1. 启动程序后等待 `S` 开始，`Q` 直接退出。
2. 调用一次 `Recorder.start()`，整个进程只维护一个 recorder。
3. 主循环中持续采集机械臂和底盘状态，并持续追加到当前 recorder。
4. 按 `Q` 退出主循环后，统一执行一次 `Recorder.commit()`。

这一版的特点是：

- 只能做一次连续采集，不能在程序内切分多个 episode。
- 不能在采集中途立即删除当前 episode。
- 不能在采集中途立即提交当前 episode。
- 键盘事件和采集循环耦合较紧，扩展踏板逻辑不方便。

## 2. 当前版本新增的目标

当前版本把采集入口改成了基于踏板控制的多 episode 采集模型，目标行为如下：

- `A`：任意时刻开始新 episode，并立即开始采集。
- `B`：采集中删除当前未保存 episode，回到空闲态，等待下一次 `A`。
- `C`：采集中立即保存当前 episode。
- `Q`：任意时刻退出程序；若当前 episode 尚未保存，则自动保存。
- 所有按键增加 `0.5s` 防连按。

## 3. 代码结构上的核心改动

### 3.1 去掉原始的单次启动模型

删除了原来的：

- `preparing()`
- `iter_action()`

这两个函数原本对应的是：

- 启动前等待 `S`
- 采集中仅监听 `Q`

它们不适合踏板驱动的多 episode 状态切换，因此被新的事件驱动结构替代。

### 3.2 新增按键监听线程 `PedalListener`

新增 `PedalListener` 后台线程，持续读取终端输入并投递命令：

- 识别按键：`A/B/C/Q`
- 所有按键统一转小写处理
- 使用 `DEBOUNCE_SECONDS = 0.5` 做防连按

这样做的目的：

- 避免把按键处理硬塞进采集主循环
- 后续更方便接入踏板映射输入
- 响应逻辑更清晰，状态切换不依赖旧的 `S/Q` 模式

### 3.3 新增 `EpisodeManager`

新增 `EpisodeManager` 负责 recorder 生命周期管理，主要职责：

- 创建 `Recorder`
- 维护当前是否有活跃 episode
- 写入动作数据
- 提交当前 episode
- 回滚当前 episode
- 程序退出时统一收尾

它把原本“一个 `Recorder` 用到底”的模型，改成了“按 episode 管理 recorder 会话”的模型。

## 4. 采集流程上的行为变化

### 4.1 支持程序内多次开始新 episode

`A` 触发后：

- 如果当前处于空闲态，则直接创建新的 recorder 并开始采集。
- 如果当前已经在采集，则先提交当前 episode，再立即开启下一个 episode。

这样可以在不重启程序的情况下连续采集多个 episode。

### 4.2 `B` 改为只删除当前 episode，不自动重开

当前最终行为是：

- `B` 只对“当前活跃且未保存”的 episode 执行 `rollback()`
- 删除完成后回到空闲态
- 不会自动开始新的 episode
- 必须等待下一次 `A` 才重新开始采集

这是相对最初修改版又额外收紧的一点，目的是让“删除”和“重开”两个动作分离，避免误触 `B` 后马上重新录制。

### 4.3 `C` 支持采集中即时提交

`C` 触发后：

- 对当前活跃 recorder 立即执行 `commit()`
- 当前 episode 保存完成后回到空闲态
- 等待下一次 `A` 再开启新 episode

这实现了“采集中途立即落盘并生成 episode 文件夹”的需求。

### 4.4 `Q` 退出时自动保存未保存数据

程序退出时在 `finally` 中统一调用：

- `episode_manager.close(save_active=True)`

因此如果退出前存在尚未提交的活跃 episode：

- 有数据则自动 `commit()`
- 无数据则不生成空 episode

这部分是为了满足“防丢数据”的要求。

## 5. 采集与控制解耦

当前版本新增了后台采样线程 `sampling_loop()`：

- 采样线程持续调用 `robot.get()`
- 若当前存在活跃 episode，则把采样结果写入 recorder
- 若当前为空闲态，则采样结果直接忽略，不写盘

这样的收益是：

- 采集逻辑和按键控制逻辑解耦
- episode 切换不再依赖旧的阻塞式主循环结构
- 程序组织更适合后续继续扩展

## 6. 常量和状态控制

新增了几个关键常量：

- `COMMAND_KEYS = {"a", "b", "c", "q"}`
- `DEBOUNCE_SECONDS = 0.5`
- `SAMPLE_INTERVAL = 0.005`

用途分别是：

- 明确允许的踏板/退出指令集合
- 统一防误触
- 控制后台采样频率

## 7. 未改动的部分

这次修改没有改变以下内容：

- `Robot.get()` 的数据内容和结构
- 双臂状态、夹爪、底盘速度的采样字段
- `Meta / RealSense / Recorder` 的外部依赖接口
- 原有的相机序列号、端口、底盘地址等配置来源

也就是说，这次改动聚焦在“采集流程控制”，不是在改数据格式或硬件通信层。

## 8. 当前版本总结

相对原始版本，本次改动本质上完成了三件事：

1. 把单次 `S/Q` 采集改成了多 episode 的踏板驱动模型。
2. 把 recorder 从“全局单实例”改成了“按 episode 生命周期管理”。
3. 增加了即时提交、即时丢弃、防连按、退出自动保存这几项采集安全机制。

如果后续继续扩展，可以优先考虑：

- 把 README 中旧的 `S/Q` 说明同步更新
- 如果踏板输入最终不是标准键盘输入，可把 `getch()` 替换成对应设备事件读取
- 如果需要更严格的“立刻响应”，可以继续评估把 `robot.get()` 拆成更细的非阻塞式读取
